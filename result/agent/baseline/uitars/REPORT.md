## 四、失败模式分析

### 4.1 失败原因分类

| 失败原因 | 任务数 | 说明 |
|----------|--------|------|
| Docker 容器无响应 | 40 | 截图获取失败，任务未能启动（vs_code 16 个，os 12 个，thunderbird 7 个，vlc 5 个）|

### 4.2 失败任务步数分布（272 个有步数记录的失败任务）

| 步数区间 | 任务数 | 占比 |
|----------|--------|------|

> 35% 的失败任务在最后 10 步内结束，说明大量任务在步数边界附近放弃。

### 4.3 典型失败模式

**模式一：Docker 崩溃（后期补跑）**

后期补跑（阶段 4）中 vs_code 全部 16 个补跑任务 Docker 容器无响应，均未进行评测。os、thunderbird、vlc 也有大量类似情况，是本次实验最主要的数据损失原因。

**模式二：模型频繁放弃（FAIL）**

libreoffice_calc 有 31 个 FAIL（共 47 个任务），multi_apps 有 44 个 FAIL（共 101 个任务）。这类任务通常涉及复杂公式、跨应用操作或网络依赖，模型在多次尝试失败后主动放弃。

**模式三：DONE 后评测失败**

chrome 有 23 个 DONE 但得 0 分的任务（占 chrome 总数 50%），主要是需要访问互联网的任务（查航班、查酒店、浏览商品），沙箱网络隔离导致系统性失败。

**模式四：步数耗尽**

libreoffice_writer（7/23 耗尽步数）和 libreoffice_impress（8/47）的步数耗尽比例较高，反映这类需要精确 GUI 操作的任务对步数消耗大。

## 六、与早期版本对比

| 实验 | 成功率 | max_steps | 模型接口 | 推理方式 |
|------|--------|-----------|----------|----------|
| results_uitars_full（早期）| 1.33% | 15 | doubao API | 远程 API |
| **results_uitars_repro_1（本次）** | **10.8%** | 50 | qwen25vl | 本地 vLLM |

成功率从 1.33% 提升至 10.8%（约 8 倍），主要原因：

1. **模型接口修正**：从 doubao 格式改为原生 qwen25vl 格式，模型能正确理解图文输入
2. **步数上限提升**：从 15 步放宽到 50 步
3. **本地 vLLM 推理**：避免 API 超时和代理拦截

---

## 七、结论

1. UI-TARS-1.5-7B 在正确配置下可达 **10.8%**（40/369 完全通过）。
2. 最强应用为 **GIMP（42.3%）和 Chrome（23.9%）**，均为视觉目标明确、操作路径浅的任务。
3. **vs_code 实际仅 1/23 = 4.3%**，——后期补跑时 Docker 全部崩溃，16 个任务未能评测。
4. **multi_apps 完全失败（0/101）**，跨应用长链任务超出当前模型能力。
5. 56 个任务因 Docker 崩溃或环境初始化失败未能评测。
